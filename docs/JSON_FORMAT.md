# Правила составления JSON для генератора видео-чата

Документ описывает формат входного файла переписки. Файл передаётся в рендер через `--input`, веб-интерфейс (`./run.sh ui`) или лежит в `public/conversation.json` для предпросмотра.

**Кодировка:** UTF-8.  
**Формат:** валидный JSON (объект в корне, не массив).

---

## 1. Структура файла

```json
{
  "contactName": "Анна",
  "contactStatus": "в сети",
  "contactAvatar": "avatar.svg",
  "wallpaper": "default",
  "myName": "Я",
  "emojiAliases": {
    "unicorn": "🦄"
  },
  "messages": [
    {
      "author": "them",
      "text": "Привет :wave:",
      "typingMs": 1500,
      "pauseBeforeMs": 600,
      "sentAt": "12:31"
    },
    {
      "author": "me",
      "text": "Привет! :smile:",
      "typingMs": 1200,
      "pauseBeforeMs": 500,
      "sentAt": "12:31"
    }
  ]
}
```

| Поле | Обязательно | Тип | По умолчанию | Описание |
|------|-------------|-----|--------------|----------|
| `messages` | **да** | массив | — | Список сообщений (минимум 1). Порядок = порядок появления в видео. |
| `contactName` | нет* | строка | `"Contact"` | Имя собеседника в шапке чата. |
| `contactStatus` | нет | строка | `"в сети"` | Статус в шапке, когда собеседник **не** печатает. |
| `contactStatusTyping` | нет | строка | `"печатает…"` | Статус в шапке **во время** анимации набора у `them` (автоматически). |
| `contactAvatar` | нет | строка | `"avatar.svg"` | Файл аватара в папке `public/`. |
| `wallpaper` | нет | `"default"` \| `"dark"` | `"default"` | Тема UI чата: светлая (`default`) или тёмная (`dark`). В веб-интерфейсе (`./run.sh ui`) тему можно выбрать переключателем — она подставится при рендере и попадёт в сохранённый JSON. |
| `myName` | нет | строка | `"You"` | Имя «вашего» профиля (в JSON есть, в UI личного чата не показывается). |
| `emojiAliases` | нет | объект | — | Дополнительные shortcodes (см. раздел 4). |
| `music` | нет | объект | см. README | Фоновая музыка: `enabled`, `src` (путь в `public/`), `volume` (0–1). |
| `outro` | нет | объект | — | Заставка «Подпишись»: `enabled`, `text`, `pauseBeforeMs`, `durationMs`. |
| `sounds` | нет | объект | — | Пути и громкость SFX (входящие, исходящие, набор). |
| `timing` | нет | объект | см. §2.2 | Коэффициенты авто-расчёта пауз и набора по длине текста. |

\*Технически поле необязательно, но для осмысленного видео `contactName` нужно указать явно.

## 2. Сообщения (`messages[]`)

Каждый элемент массива — одно сообщение в диалоге.

| Поле | Обязательно | Тип | По умолчанию | Описание |
|------|-------------|-----|--------------|----------|
| `author` | **да** | `"me"` \| `"them"` | — | Кто отправил сообщение. |
| `text` | *условно* | строка | `""` | Текст или подпись к фото. Обязателен, если нет `image`. Поддерживает emoji и shortcodes. |
| `image` | нет | строка | — | **Локальный путь** в `public/` (напр. `images/photo.jpg`) или **URL** (`https://…`). Фото без `text`, или фото + подпись в `text`. |
| `imagePrompt` | нет | строка | — | Промпт для **первичной генерации** кадра (OpenRouter / ChatGPT). |
| `imageEditPrompt` | нет | строка | — | Правки к **уже готовому** изображению (кнопка «Применить правки» в UI). |
| `typingMs` | нет | число | *авто* | Длительность набора, мс. Если не указано — **считается по длине текста**. |
| `pauseBeforeMs` | нет | число | *авто* | Пауза перед набором, мс. Если не указано — **считается по длине текста**. |
| `sentAt` | нет | строка | `"12:34"` | Время в углу пузыря (любая строка, напр. `"18:02"`). |

### 2.1. Значения `author`

| Значение | В интерфейсе | Анимация |
|----------|--------------|----------|
| `them` | Входящее: белый пузырь слева | Пауза → индикатор «печатает…» (точки) → появление сообщения → звук входящего |
| `me` | Исходящее: зелёный пузырь справа | Пауза → посимвольный набор в поле ввода → отправка → звук исходящего |
| `me` / `them` + `image` | Пузырь с фото | Короткая пауза; у `them` — точки «печатает…», у `me` без набора в поле (если нет подписи). Время — в углу фото или под подписью. |

Сообщения воспроизводятся **строго по порядку** в массиве `messages`. Нельзя «вставить» сообщение в прошлое — только изменить порядок элементов.

### 2.2. Тайминг одного сообщения (авто по длине)

Для каждого сообщения на временной шкале:

```
[ pauseBeforeMs ] → [ typingMs ] → [ показ пузыря ] → [ postReveal ] → следующее сообщение
```

После последнего сообщения к ролику добавляется **хвост ~1.5 с**.

Если в JSON **не заданы** `typingMs` и `pauseBeforeMs`, длительности считаются автоматически от числа символов (emoji = 1 символ) и числа строк.

**Формулы по умолчанию:**

| Фаза | `them` (входящие) | `me` (исходящие) |
|------|-------------------|------------------|
| Пауза перед набором | `350 + 8×символы + 120×(строки−1)` мс | то же |
| Набор | `700 + 28×символы` мс (точки «печатает…») | `280 + 48×символы` мс (посимвольно в поле ввода) |
| После появления пузыря | `550 + 12×символы` мс | то же |

Все значения ограничены минимумом/максимумом (например, набор: 450–9000 мс).

**Пример:** короткое «Ок» (~2 символа) — ~0.5 с набора; длинное сообщение на 80 символов — ~3–4 с набора у `me`.

**Ручная подстройка** — укажите в сообщении:

```json
{ "author": "them", "text": "Думаю…", "pauseBeforeMs": 2000, "typingMs": 2500 }
```

**Глобальная настройка скорости** — блок `timing` в корне JSON:

```json
{
  "timing": {
    "meTypingPerCharMs": 55,
    "themTypingPerCharMs": 32,
    "pausePerCharMs": 10
  },
  "messages": [ ... ]
}
```

### 2.3. Сообщения с изображением

#### Локальный файл

Положите картинку в `public/` (обычно `public/images/`) и укажите путь **без** ведущего `/`:

```json
{
  "author": "them",
  "image": "images/photo.jpg",
  "text": "Смотри :eyes:",
  "sentAt": "14:20"
}
```

Только фото (без подписи):

```json
{
  "author": "me",
  "image": "images/screenshot.png",
  "sentAt": "14:21"
}
```

Демо в репозитории: `public/images/demo.svg`.

**Форматы:** jpg, jpeg, png, webp, gif, svg.

При сборке видео: если файл **не прикреплён** (пути нет на диске), картинка **не попадает** в ролик — остаётся только `text`, если он есть. Сообщение **только с `image`** без файла **полностью пропускается**.

**Генерация через OpenRouter:** если задан **OpenRouter** (`OPENROUTER_API_KEY`), промпт к кадру собирается по **полной переписке** (ChatGPT); при наличии **предыдущих фото** в чате (файлы в `public/images/`) LLM получает их как референс и требует преемственность (тот же вагон, стиль). При генерации 2-го и следующих кадров **последнее предыдущее фото** передаётся в image-to-image. Первый кадр — без референса.

Пример:

```json
{
  "author": "me",
  "image": "images/last_carriage_01_empty_wagon.png",
  "text": "Вагон пустой 😟",
  "imagePrompt": "Пустой вагон электрички, ряды сидений, никого в кадре, тревожная атмосфера, вид с телефона"
}
```

#### Ссылка (URL)

Можно указать прямую ссылку на картинку — при рендере она **скачивается** в `public/images/`:

```json
{
  "author": "them",
  "image": "https://example.com/photos/ref.jpg",
  "text": "Референс"
}
```

- Поддерживаются `http://` и `https://`
- Лимит размера при скачивании: **12 МБ**
- После успешного рендера в сохранённом JSON путь обычно заменяется на локальный (`images/…`)

#### Веб-интерфейс: прикрепление картинок

В `./run.sh ui` под полем JSON появляется блок **«Изображения в переписке»** (обновляется при редактировании JSON):

| Статус | Что значит | Действие |
|--------|------------|----------|
| Файл на месте | Путь в `public/` существует | Превью, можно заменить файлом |
| Нужно скачать или прикрепить | В JSON указан URL | **Скачать по ссылке** или **Прикрепить файл** |
| Файл не найден | Локальный путь, файла нет | **Прикрепить файл** (сохранится по этому пути или в `images/`) |

После прикрепления или скачивания поле `image` в JSON **обновляется автоматически** на локальный путь.

#### Генерация через OpenRouter (веб-UI)

Если в `.env` (корень проекта или `docs/.env`) задан ключ:

```env
OPENROUTER_API_KEY=ваш_ключ
OPENROUTER_TEXT_MODEL=openai/gpt-5.4
OPENROUTER_IMAGE_MODEL=google/gemini-2.5-flash-image
OPENROUTER_STORY_IMAGE_MODEL=google/gemini-2.5-flash-image
OPENROUTER_IMAGE_ASPECT_RATIO=4:3
OPENROUTER_IMAGE_SIZE=1K
```

По умолчанию картинки для пузыря чата: **4:3** (~672×504 в видео), **1K**. Текст диалогов и промпты к кадрам — через `OPENROUTER_TEXT_MODEL`; генерация изображений — через `OPENROUTER_IMAGE_MODEL`.

В блоке диалога картинки генерируются через **OpenRouter (ChatGPT)**. Автопромпт (LLM) формирует сцену по полной переписке; при генерации промпт подставляется автоматически, если поле пустое.

1. Промпт берётся из **`imagePrompt`** или собирается автоматически по переписке.
2. OpenRouter создаёт картинку (**4:3**, 1K — размер под пузырь в чате), файл в `public/images/`.
3. Поле `image` в JSON подставляется автоматически.

При рендере URL в `image` по-прежнему скачиваются на сервер; сгенерированные локальные файлы используются как есть.

**Ограничения:** генерация асинхронная (обычно 10–120 с). Баланс и лимиты — в [личном кабинете OpenRouter](https://openrouter.ai/). Пока ключ не настроен — **«Из буфера»** / **«Скачать по ссылке»**.

### 2.4. Переносы строк

В `text` можно использовать `\n` для многострочных сообщений:

```json
{
  "author": "me",
  "text": "Первая строка\nВторая строка :fire:"
}
```

## 3. Оформление чата

### 3.1. `wallpaper`

| Значение | Вид |
|----------|-----|
| `default` | Светлый фон (как WhatsApp) |
| `dark` | Тёмный фон |

### 3.2. `contactAvatar`

Путь к файлу **относительно папки `public/`**:

```json
"contactAvatar": "avatar.svg"
```

```json
"contactAvatar": "photos/anna.jpg"
```

Файл `public/photos/anna.jpg` должен существовать до рендера. Если файл не найден, рендер может завершиться с ошибкой.

Если аватар не нужен — можно оставить значение по умолчанию: в шапке покажется **первая буква** `contactName`.

### 3.3. `contactStatus` и `contactStatusTyping`

- `contactStatus` — обычный статус («в сети», «был(а) недавно»). Показывается всё время, **кроме** фазы набора входящего сообщения.
- `contactStatusTyping` — подпись в шапке, пока `them` «печатает» (синхронно с точками в чате). После отправки сообщения снова показывается `contactStatus`.

Не ставьте `"печатает…"` в `contactStatus` — иначе после конца переписки статус останется «печатает».

## 4. Emoji и смайлы в `text`

Поддерживаются **три способа** (можно комбинировать в одной строке).

### 4.1. Unicode напрямую

```json
"text": "Привет 👋 Всё огонь 🔥"
```

Рекомендуется для точного контроля над символом.

### 4.2. Shortcodes `:код:`

Формат: двоеточие, имя кода (латиница, цифры, `_`, `+`, `-`), двоеточие.

```json
"text": "Супер :fire: :check: :thumbsup:"
```

- Регистр **не важен**: `:FIRE:` = `:fire:`
- Неизвестный код **не заменяется** (`:unknown:` останется как есть)
- Полный встроенный список: [`public/emoji-shortcodes.json`](../public/emoji-shortcodes.json)

**Встроенные коды (выборка):**

| Код | Emoji | Код | Emoji |
|-----|-------|-----|-------|
| `:smile:` | 😊 | `:wave:` | 👋 |
| `:heart:` | ❤️ | `:fire:` | 🔥 |
| `:thumbsup:` | 👍 | `:check:` | ✅ |
| `:laugh:` | 😂 | `:cry:` | 😢 |
| `:thinking:` | 🤔 | `:party:` | 🎉 |
| `:movie:` | 🎬 | `:rocket:` | 🚀 |
| `:eyes:` | 👀 | `:hundred:` | 💯 |

### 4.3. Текстовые аналоги

ASCII-смайлы заменяются на emoji при загрузке JSON:

| В тексте | Результат |
|----------|-----------|
| `:)`, `:-)`, `=)` | 🙂 |
| `:(`, `:-(` | 🙁 |
| `:D`, `:-D` | 😄 |
| `;)`, `;-)` | 😉 |
| `<3` | ❤️ |
| `:'(` | 😢 |
| `:/`, `:-/` | 😕 |

Пример:

```json
"text": "Отлично :) спасибо <3"
```

**Важно:** длинные последовательности обрабатываются первыми (`:-)` раньше `:)`). Не используйте shortcodes внутри слов без двоеточий с обеих сторон.

### 4.4. Свои коды (`emojiAliases`)

```json
{
  "emojiAliases": {
    "unicorn": "🦄",
    "logo": "✨"
  },
  "messages": [
    { "author": "them", "text": "Смотри :unicorn: :logo:" }
  ]
}
```

- Ключ — имя без двоеточий (в тексте пишется `:unicorn:`)
- Значение — один или несколько Unicode-символов (emoji)
- Свои коды **перекрывают** встроенные с тем же именем

### 4.5. Порядок обработки

1. Замена `:shortcode:` (встроенные + `emojiAliases`)
2. Замена текстовых аналогов (`:)`, `<3`, …)
3. Unicode в строке не изменяется

Подстановка выполняется **до рендера**; в видео уже отображается итоговый текст (включая анимацию набора у `me`).

## 6. Проверка и типичные ошибки

### Валидация

При неверном JSON или полях рендер завершится с ошибкой (Zod). Частые причины:

| Ошибка | Причина |
|--------|---------|
| `messages` must contain at least 1 element(s) | Пустой массив `messages` |
| Укажите text и/или image | Пустые `text` и `image` в одном сообщении |
| Нет сообщений для рендера… | Все сообщения пустые или только с неприкреплённым `image` |
| Invalid enum value | `author` не `me` и не `them` |
| Number must be greater than or equal to 200 | `typingMs` < 200 |
| Invalid enum value | `wallpaper` не `default` / `dark` |

Проверка типов локально:

```bash
npm run lint
```

### Частые ошибки

1. **Невалидный JSON** — лишняя запятая после последнего элемента, одинарные кавычки вместо двойных.
2. **Путь к аватару или картинке** — файл не лежит в `public/` (для URL используйте рендер через UI или дождитесь авто-скачивания).
3. **Слишком короткий `typingMs`** — меньше 200 мс (ограничение схемы).
4. **Ожидание имени над пузырём** — в личном чате имена в сообщениях не выводятся, только в шапке (`contactName`).
5. **Shortcode без двоеточий** — нужно `:fire:`, а не `fire`.
6. **Конфликт `:) ` с shortcodes** — в редких случаях `:` в тексте может мешать; для надёжности используйте Unicode или shortcodes.

## 7. Запуск с вашим JSON

```bash
# Веб-UI: тема, музыка, прикрепление изображений из JSON
./run.sh ui
# → http://localhost:3333

# Локально (URL в image скачиваются автоматически)
node scripts/render.mjs --input ./my-chat.json --output ./out/my-chat.mp4

# Docker (сначала один раз: ./run.sh build — Chrome кэшируется в образе)
./run.sh render --input ./my-chat.json --output ./out/my-chat.mp4

# Предпросмотр (подставьте путь в Remotion Studio → Props → conversation)
npm run dev
```

Готовые примеры в репозитории:

- [`public/conversation.json`](../public/conversation.json) — демо по умолчанию
- [`my-chat.json`](../my-chat.json) — тестовый сценарий с emoji
